3. System Context

The system contains the following Dockerized components:

  • Frontend: Nginx-based gateway and static frontend on port 8080.
  • auth-service: Firebase authentication verification on port 4000.
  • product-service: Product and category API backed by MongoDB on port 4001.
  • order-service: Order creation and order history API backed by MongoDB on port 4002.
  • user-service: Profile/project API and internal calls to order-service on port 4003.
  • chat-service: User/admin messaging API backed by MongoDB on port 4004.
  • Prometheus: metrics and alerts on port 9090.
  • Grafana: dashboard visualization on port 3000.
  • Terraform: infrastructure provisioning for the Docker host and firewall rules.

4. Automation Implementation

4.1 Automated deployment

The full stack can be deployed using one PowerShell command:

.\scripts\start-stack.ps1

This script validates configuration first and then runs:

docker compose up -d --build

4.2 Configuration validation

The pre-deployment validation script checks the required configuration keys:

  • FIREBASE_PROJECT_ID
  • FIREBASE_CREDENTIALS_HOST_PATH
  • CORS_ORIGINS
  • MONGO_URL

It also rejects the known incident simulation value mongodb://wrong-mongodb:27017/shop, which helps prevent a repeat of the earlier database misconfiguration incident.

4.3 Health checks and restart policies

All backend services expose /health and /metrics. Docker Compose includes health checks and restart: unless-stopped policies. This gives the system automatic failure detection at the container level and restart recovery when a service exits unexpectedly. 

healthcheck:
  test: ["CMD-SHELL", "node -e ... /health ..."]
  interval: 10s
  timeout: 5s
  retries: 3
  start_period: 20s

restart: unless-stopped

6. Monitoring and Alerting

Prometheus is configured with a 5-second scrape interval and collects metrics from each backend service.

rule_files:
  - /etc/prometheus/alerts.yml

Implemented alert rules:

  • ServiceDown: Prometheus cannot scrape a service.
  • ServiceUnhealthy: a service health metric reports unhealthy state.
  • HighErrorRate: more than 5 percent recent HTTP responses are errors.
  • HighLatency: p95 latency is above 1 second.
  • HighProcessCpu: a service process uses more than 0.8 CPU cores.
  • HighProcessMemory: a service process uses more than 300 MB resident memory.
  • ServiceRestartedRecently: a service process start time changed within 10 minutes.

7. Capacity Planning

7.1 Load simulation method

A Node.js load-test script was added to generate concurrent requests and report request rate, success count, failure count, average latency, p95 latency, and p99 latency. 

node .\scripts\load-test.mjs --url http://localhost:4002/health --requests 1000 --concurrency 50

The load test focused on order-service because this service is the main capacity risk: it owns order workflows and depends on the database layer. A health endpoint was used for repeatable local load generation without requiring live Firebase user tokens.

Production-like testing of POST /api/orders and GET /api/orders/my requires valid Firebase ID tokens and realistic product/user data. For this assignment, the /health endpoint was used as a controlled baseline to measure service availability, request handling, latency, and observability behavior without introducing authentication-token setup as a separate variable. Therefore, the result should be interpreted as a local baseline capacity test, not as the absolute production capacity of the order workflow.

8. Bottleneck and Capacity Analysis

The current local load test did not saturate CPU, memory, or latency thresholds. However, the architecture still has clear capacity risks:

  • order-service: highest business-critical capacity risk because it owns order workflows and depends on MongoDB.
  • MongoDB: shared dependency for product-service, order-service, and chat-service. A database slowdown can degrade several services at once.
  • Single-container services: each backend service runs as one container, so a single service instance limits horizontal capacity.
  • Frontend gateway: Nginx currently routes to single service names; future multi-instance routing should use an orchestrator or load balancer.

The main conclusion is that the system is reliable under the tested local health-check load, but production-like capacity testing should use authenticated order creation and order history requests with realistic user data.

10. Scaling Strategy

10.1 Horizontal scaling

The first horizontal scaling target should be order-service. Future deployment should run multiple order-service replicas and place a load balancer or orchestrator in front of them. 

order-service-1
order-service-2
order-service-3

Possible implementation options:

  • Docker Swarm service replicas with ingress routing.
  • Kubernetes Deployment and Service objects.
  • Nginx upstream load balancing for several backend instances.

10.2 Vertical scaling through Terraform

Terraform already owns VM infrastructure definitions. If load increases, the VM type can be increased through the Terraform variable machine_type. 

machine_type = "e2-medium"

Future larger option:

machine_type = "e2-standard-2"

10.3 Database optimization

  • Add indexes for common order queries, especially user order history by userId and creation time.
  • Tune MongoDB CPU and memory resources or move the database to a managed service.
  • Use connection pooling and sensible timeout settings.
  • Separate read-heavy and write-heavy workloads if request volume grows.

12. Limitations and Next Steps

The current implementation demonstrates automation, alerting, load simulation, and recovery. It does not fully implement production auto-scaling. Horizontal scaling, Kubernetes-based auto-scaling, and managed database scaling are proposed strategies rather than completed runtime features.

The load test is also intentionally limited to the /health endpoint. Authenticated order endpoints require Firebase ID tokens, so a production-like test would need a token generation or login flow, seeded product IDs, and realistic user scenarios.

Recommended next steps:

  • Run an authenticated load test against real order creation and order history endpoints.
  • Add container-level CPU and memory metrics with cAdvisor or Docker metrics exporter.
  • Run multi-replica order-service behind a load balancer.
  • Move database capacity testing to realistic product/order datasets.
  • Integrate alert delivery through Alertmanager, email, Telegram, Slack, or another notification channel.

13. Conclusion

The system now includes practical SRE automation and observability improvements: validated configuration, health checks, automatic restart policies, Prometheus alert rules, Grafana dashboards, load testing, and recovery evidence. Capacity testing shows that the local order-service health endpoint remains stable under concurrent traffic with zero failures, while the architecture analysis identifies order-service and the shared database as the main capacity risks for future production-like workloads.
